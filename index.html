01一句话主张

模型怎么调、链路怎么改，方法千千万，但重要的是先把「尺子」做对。
用户效用度量，是地图也是方向盘——度量错了，优化越使劲，跑得越偏。

02从一个最简单的 case 说起：用户心里在打分

一个人在搜索框敲下："预算 3000 想给爸妈换手机，要屏幕大、续航久、好上手，选哪款？"

他要的不是参数表

而是和我处境一样的人（给老人买、图省心）真实买过什么、踩过哪些坑——借真实成交和评价做自己的购买决策。

系统一两秒内做三件事

召回真实成交 / 评价 / 导购笔记 → LLM 组织成贴合处境的推荐 → 呈现；他在心里默默打分，再用脚投票（下单 / 加购 / 退货）。

那个"分"我们看不见、没法直接优化——只能用一个模型去近似它，这就是 Reward Model。

03于是"优化 AI 产品"拆成三个核心问题

RM 是我们对"用户那把尺子"的近似，它就是模型做 RL 时的优化导向。

1尺子怎么刻

人类对内容的效用 / 质量如何度量？维度、档位、怎么合成一个分。

2怎么教给模型

模型如何拟合这把尺子（RM）？让计算机能替我们打出这个分。

3尺子对不对

线上真实用户认不认？后验行为与 AB 回头校准这把尺子。

全篇就围绕这三件事：刻尺子 → 教模型 → 用户验收。

06效果度量是地图和方向盘（一）：通用五维度

人类怎么判断一个答案好不好——先有这五条，再谈场景化。
维度通俗讲电商 badcase医疗 badcase
满足需求有没有解决用户真正想问的问"平价防晒"列一堆贵价大牌问"能不能停药"只甩说明书、不给判断
相关性有没有跑题、过度延展问防晒，答成护肤全流程科普问感冒用药，答成疾病百科大全
真实有用符不符合事实、过不过期、缺不缺角推荐已停产 / 改了配方的旧产品引过期指南、剂量 / 单位写错
信任感用户信不信、推理连不连贯清一色"网红同款"软广该提示就医却含糊、编造把握
阅读体验啰不啰嗦、结构清不清晰流水账，得自己扒出推荐哪款满屏免责声明，真正建议被淹没
通用维度只是起点——真正的难点在场景化、怎么刻画、怎么 merge。

07效果度量（二）· 场景化：用户预期决定尺子怎么刻

同一个答案，换个场景好坏完全不同——因为用户的预期不一样。
电商导购搜医疗问诊搜通用知识搜内容种草搜
用户要的买到适合自己的对的东西安全对症的处置 + 该不该就医一个客观正确的事实灵感 + 同类人的真实体验
"真实"指真实销量 / 评价、非刷单合指南 / 共识、有循证、不臆断符合事实、无幻觉真人真事、非软广
"权威"来自真实成交 / 同类买家评价临床指南 / 专家共识 / 权威机构官方 / 专家用过的同类人
独有维度适配（肤质/机型）· 性价比 · 避坑安全性（红旗/禁忌）· 提示就医 · 不确定 abstain完整 · 准确种草感 · 人群匹配
一票否决刷单虚假评价 / 错价错参 / 推无货误导用药 / 漏红旗 / 超范围下结论事实错误软广 / 编造
没有"通用最优"，只有场景最优——脑中先有最优答案，才能做出超预期。

08效果度量（三）· 怎么刻画一个维度：四种方法

关键看它有多可稳定验证，从易到难分四类。
类型（易 → 难）怎么打分电商例子医疗例子
有明确答案直接判定，量一下就知道商品在不在售、价格参数对不对剂量/单位对不对、是否触禁忌清单
线性可比定清 0–4 档，照档打分商品适配度（肤质/机型）、性价比分诊紧急度、科普权威度（指南/共识）
Rubric拆成多条 criteria，逐条判定这条评价是不是刷单、推荐够不够全有没有漏红旗症状、是否提示就医、有无超范围
GSB好/平/差两两对比，只比相对优劣"是不是让我更想下单""哪版更让患者放心、敢遵医嘱"
度量定义不一定要走重流程：一个有判断力的 PM + 一段 prompt，就能产出 v0。

09Rubric 是什么：维度 × 档位，不是一个抽象分

把评判拆成多条 criteria，每条配一个明确判定档，而非拍一个抽象分。

1维度（criteria）= 该看哪几条

  • 是否按处境（肤质/机型/预算/尺码）匹配？
  • 是否基于真实销量 / 评价，而非刷单 / 软广？
  • 有没有避坑 / 适用边界（缺货、版本差异）？
  • 看完能直接下单，还是"自己再看看"？

2档位（levels）= 每条怎么判

  • Green：完全符合，可采纳
  • Yellow：方向对，细节有瑕疵
  • Red：错误或有害，必须拦截
  • Underconstrained：信息不足，应主动说"不确定"
换到医疗：criteria = 是否核对剂量 / 禁忌、有无漏红旗、是否提示就医、信息不足是否 abstain；档位同样 Green / Yellow / Red / Underconstrained。
维度 × 档位拼出失败的坐标：错在哪条、错到哪一档。

10GSB 是没办法的办法：能拆成 Rubric 的，尽量别停在 GSB

GSB（好/平/差两两对比）只给一个整体偏好，代表"某种体验"，却不告诉你好在哪、错在哪——是最后才用的兜底。

为什么会卡在 GSB

  • 大概率不是事情不可拆，是缺这个领域的专家
  • 例：「美」——只会说这个好看那个不好看，因为团队里没有美学专家
  • 没人能说清标准，就只能两两对比

有专家，再主观也能拆成 Rubric

  • 真专家总能给出 criteria：「美」美在哪
  • 如：白底干净 / 45° 或斜角构图 / 留白比例⋯
  • 每条再配 Green / Yellow / Red 档位
想尽办法把 GSB 往 Rubric 转，是评测体系最重要的事之一——GSB 只该是临时兜底，不是终点。

11效果度量（四）· 多维 merge：别用加权平均

一条错误的建议，再流畅、结构再好，总分也不该高。

1硬约束（一票否决）

电商：刷单虚假好评、错价错参、推无货 / 停售品；医疗：误导用药、漏红旗、超范围下结论——分数直接压下去。

2场景调权

电商抬真实评价 / 适配 / 性价比；医疗抬安全性 / 循证 / 该就医提示；按品类 / 科室再微调权重。

3叠主观分

丰富度、结构、表达、简洁——锦上添花，但永远在硬约束之后。

线性加权是最差的选择：它会让"漂亮的错误"捞到高分。

12Failure Code：把"用户的不满"结构化

电商例：问"预算 3000 给爸妈换手机"，却推一水儿商单"网红爆款"、还都是年轻人在用 → 软广未识别 + 人群不匹配。
医疗例：问"孩子发烧能吃这个药吗"，没核对年龄 / 剂量 / 禁忌就给肯定答复 → 剂量未校验 + 该 abstain 未 abstain。

Code 是"现象"，不是"根因"

"软广"的根因可能在召回（捞进商单）、prompt（没要求排广告）、或 RM（没有软广维度）——要再往下归因。

Code 的三个用处

  • 组织评测集：用户眼里的问题解没解
  • 组织训练集：专做对抗样本
  • 做效果分析：指标掉了，哪个 Code 在作怪
好 Code：可观察、判定一致、与效果直接相关、能拿来训 RM。

13召回 / Search：答案的"原材料"，是效果的天花板

典型 garbage in, garbage out——召回上来的就不对，后面再强也救不回。

失败模式

  • 电商：召回刷单软广、人群不匹配、过期促销
  • 医疗：召回过期指南、非权威网帖、个例当通则
  • 只召一边倒好评，漏掉避坑帖 / 禁忌

理想态 & 离线手段

  • 电商：真实 · 对人群 · 不过期 · 覆盖全
  • 医疗：优先现行指南 / 共识 / 权威库
  • rerank 把最该被信任的排到前面
召真实成交 / 真实评价 / 优质导购笔记，而非刷单软文、无货链接、过期促销。

14LLM 生成：把多篇经验拧成决策建议，锚点是 Reward Model

这一步在干什么

大模型拿用户问题 + 召回笔记，综合成一段贴合处境的建议——谁适合、坑在哪、怎么选。这是用户唯一直接看到的东西。

RM = 整个优化的"方向盘"

RM 在很大程度上就是模型做 RL 时的 label：它给什么样的答案打高分，模型就往哪个方向长。

同样一批笔记，可以"帮你把决定做完"，也可以是"正确的废话"——一大半压在这一步。

15RM 训练数据：把效果度量翻译成模型能学的样本

类型怎么来价值 & 注意
人工标注高价值 case / 典型 badcase / AB diff / 仲裁样本量不一定多，但必须干净
Pairwise 偏好A vs B 谁更好 + 为什么相对序比绝对分稳；多喂 hard negative
大模型合成M 模型各生成 N 条，再挑最好的今天最重要的来源，但有系统性偏差
Code 专项某类 badcase 攒成专项集（如"软广未识别"）让模型学会识别典型错误
合成的坑：LLM 当裁判偏爱长、偏爱结构化、对事实错误不敏感——软广常写得比真帖还"专业"，绝不能自产自销，必须独立校验。

16怎么把模型推向这把尺子：SFT → 蒸馏 → DPO → RL

一条主线：从模仿标准答案，到模仿强教师，到对齐"为什么 A 比 B 好"，再到自己探索更优。
方式怎么训优点缺点 / 坑
SFT 监督微调模仿高质量答案（标准答案）简单稳、快定基线行为偏记忆、泛化弱于 RL；天花板 = 标注，不会自我超越
MOPD 蒸馏用更强教师 / Best-of-N 选优的输出当目标，蒸给学生稳、便宜；把大模型 / 采样能力平移到线上小模型天花板 = 教师；学不到教师之外；继承教师偏差
DPO 直接偏好优化拿 pairwise（A > B）直接优化，不显式训 RM、不在线采样工程简单、稳、省一个 RM纯离线、OOD 泛化弱；难做硬约束 / 多目标
RL (RLHF/PPO/GRPO)用 RM 当 reward，在线采样迭代策略探索、泛化强；对齐尺子、适配多目标贵、不稳；RM 抓到伪特征 → 过优化 / reward hacking
训练还有什么：RLAIF（AI 反馈替代人标）· 过程奖励 PRM（给推理每一步打分）· KTO / ORPO（偏好优化变体）· 课程学习 / hard example 重采样。
来源：DPO arXiv:2305.18290（NeurIPS'23）· 蒸馏 Hinton arXiv:1503.02531 · SFT vs RL 泛化 arXiv:2501.17161（ICML'25）· Reward hacking / 过优化 arXiv:2402.09345

17训练管线：先分清基模「有没有」，再决定怎么训

别一上来就训。先问：这个能力基模到底有没有——答案决定用哪种训练方式。

1过滤已具备

基模已经会的，别花训练预算重教——先测出来，从训练集里踢掉。

2高采样探边界

对剩下的大量采样（pass@k / Best-of-N）：能采到对的 → 已具备，只是激发/稳定差；怎么都采不到 → 真不具备。

3分而治之

激发/稳定问题 → RL 直接拉稳；真不具备 → SFT 冷启动注入种子，再交给 RL。

SFT 只用于 cold start：除冷启动外不做 SFT——SFT 偏记忆、会侵蚀基模已有的泛化，提升与稳定都交给 RL。

18模仿家族：SFT · 蒸馏 · LoRA ——原理、关键参数、怎么调

都是模仿已有答案的范式：快、稳、可控，但天花板 ≈ 被模仿对象，主要用来冷启动。
方式原理最重要参数 · 经验值调大 / 调小的影响
SFT 监督微调直接模仿高质量答案学习率 1e-5 ∼ 2e-5；轮数 1–3LR 高 → 遗忘/震荡；轮数多 → 记忆、丢泛化
蒸馏学更强教师的输出（序列级 KD / 拒绝采样）温度 T；教师质量T 高 → 学到更软的分布；天花板 = 教师
LoRA 低秩微调冻结主干，只训低秩增量 ∆W=BA秩 r = 8∼64；α≈2r；LR 1e-4∼3e-4r 大 → 容量↑但易过拟合；α/r 控更新幅度
QLoRA = 4-bit 量化基座 + LoRA，单卡可训；LoRA 适配器可热插拔做多任务，但极复杂能力仍需全参训练。

19硬蒸馏：一步拿到顶级能力，代价是打崩原生推理——不推荐裸用

把 SOTA 老师模型（或多模型投票）的「推理 → 结果」整段拽出来，直接 SFT 进基模。

好处

  • 一步到位：瞬间获得接近老师的顶级能力
  • 工程极简、见效最快

坏处（为什么不推荐裸用）

  • 强灌老师的 token，基模原生推理方式被打崩
  • 学到「形」丢了「神」，OOD 一塌；后续 RL 也难救
怎么既拿能力又保住推理？没有标准答案，三条伤害更小的路径：
1 少量 SFT 做 cold start + RL 激发：只灌极少老师数据起步，能力靠 RL 逼出来。RL 本质是把 pass@k 拉成 pass@1——只要采样里激发过一次对的，就能放大。
2 自补 CoT：正确答案摆在那，但让待训基模自己补出推理链（拒绝采样留对的），用自己的推理方式而非老师的。
3 两者结合：cold start 起步 → 自采样/自补 CoT 喂回 → RL 精修，全程少碰老师的原始 token。

20On-Policy 蒸馏（OPD）：卡在 SFT 和 RL 中间的第三条路

让学生先按自己策略生成轨迹，教师在学生真实会到达的状态上给稠密反馈——纠正的正好是它自己真会犯错的地方。

定位：模仿与 RL 的中间层

  • 比 SFT 更贴近推理时真实分布（学自己的轨迹，不是老师的理想路径）
  • 比 RL 的稀疏 outcome reward 给得更稠密（教师逐 token 反馈）

正好补上「硬蒸馏崩推理」

  • 硬蒸馏：在教师理想路径上学 → 打崩原生推理
  • OPD：在学生自己轨迹上学 → 保住推理方式
  • 即上节「保留推理」的正式范式
工业拐点（Qwen3）：同样从 8B checkpoint 出发，OPD 在数学/代码上优于直接 RL，且 GPU 小时约只要 1/10。

21OPD 全景时间线（2023 → 2026）：从奠基到系统化

阶段关键进展代表工作
2023 · 范式奠基立起 on-policy 蒸馏范式GKD、MiniLLM
2024 · 目标函数细化打磨 KL 选法（forward/reverse/skewed）怎么选才不发散DistiLLM
2025 · 工业化配方Qwen3 第一次写成正式配方（8B、1/10 GPU 小时）Qwen3 报告、AlignDistil
2026H1 · 角色扩展多教师整合 / 自蒸馏 / 多模态 / prefix-only 砍成本到几十分之一MoPD、Video-OPD、OPCD
2026H1 · 理论 + 系统成败两条件；后缀监督衰减、可信 region、跨 tokenizer、异步容旧Rethinking / Lightning OPD
成败两条件（Rethinking OPD）：① 师生思维模式要兼容；② 教师必须真带来学生没见过的新能力。

22OPD 最该记的几点 + 入门路线

六个要点速览

  • 1 定位：监督比 outcome reward 稠密、比 SFT 更近真实状态
  • 2 工业拐点：8B 同起点，胜直接 RL、GPU 约 1/10
  • 3 成败两条件：思维兼容 + 教师带来新能力
  • 4 选择性监督：前 30% token 含近全部有效信号，只学前缀 FLOP 降到 1/2∼1/47
  • 5 不必白盒：拿不到 teacher logits 也能蒸（GAD→ROPD→OmniOPD）
  • 6 已扩自蒸馏/多模态：无更大教师也成，扩到 vision/video/agent

入门路线

  • GKD + MiniLLM 建直觉
  • Qwen3 看真实流水线里的位置
  • Rethinking + Lightning OPD：连「为什么有效」与「怎么降本」
  • 两篇综述收口（分类框架 + 开放问题）
开放问题：学生偏离教师后，哪些后段监督仍有行动价值。来源：「On-Policy 蒸馏后训练全景综述」Vincent｜AIGC（小红书），含 38 篇 arxiv，本地存 OPD-综述/。

23离线偏好优化（off-policy）：拿现成偏好数据，稳而省

不在线采样，直接拿固定的 A > B 偏好对优化——工程最简单、最稳，适合冷启动后快速对齐。
方法一句话最重要参数 · 经验值影响
DPO用 A > B 对直接优化，隐式 RMβ = 0.1（0.01∼0.5）；LR 5e-7∼1e-6β 小 → 偏离参考远、易退化；大 → 几乎不学
IPO平方损失替 logsigmoid，修过拟合正则 τ缓解 DPO 对偏好对的过拟合
KTO用单条好/坏信号，无需成对β数据更易得（不必配对）
ORPO无参考模型，SFT + 赔率比一步到位λ（偏好项权重）省一个参考模型；λ 调偏好强度
SimPO无参考、长度归一奖励β、γ（margin）省参考模型；γ 控正负样本间隔
共性：省、稳、好部署；但只学得到数据里见过的，OOD 泛化弱、不会探索。
来源：DPO arXiv:2305.18290 · IPO 2310.12036 · KTO 2402.01306 · ORPO 2403.07691 · SimPO 2405.14734

24在线 RL（on-policy）：自己采样、自己迭代，泛化最强也最贵

用 RM 当奖励，模型自己生成、自己评分、自己改——能探索出数据外的更优解，但贵且不稳。
方法一句话最重要参数 · 经验值影响
PPOactor + critic，裁剪式更新clip ϵ = 0.2；KL 系数；GAE λ = 0.95ϵ 大 → 不稳；KL 小 → reward hacking，大 → 学不动
GRPO去掉 critic，组内相对优势组大小 G = 8∼64；KL β；ϵG 大 → 优势估计稳但更贵；省一个价值网络
RLOOREINFORCE + 留一基线样本数 k；学习率简单省显存，适合中小规模
REINFORCE++REINFORCE + PPO 技巧（KL/归一/裁剪）KL 系数；优势归一化比 GRPO 更轻，稳定靠归一化
共性：探索强、泛化强；但在线采样贵、易不稳、会 reward hacking——KL 是安全带。
来源：PPO arXiv:1707.06347 · GRPO（DeepSeekMath）2402.03300 · RLOO 2402.14740 · REINFORCE++（OpenRLHF, 2025）

25GRPO：砍掉 critic，用「一组答案互相比」当基线——便宜、稳、为推理而生

为什么这么火：省掉和策略一样大的价值网络，显存腰斩；可验证奖励 + GRPO = DeepSeek-R1 的推理配方。

1原理

同一道题采一组 G 个答案，用组内奖励均值当基线：

Âᵢ = (rᵢ − mean(r)) / std(r)

省掉 PPO 的 critic。

2关键参数

  • 组大小 G = 8∼64
  • KL 系数 β：安全带
  • clip ϵ、采样温度

3最佳实践

  • 组内归一化奖励
  • 奖励可验证（数学/代码）最香
  • G 大 → 稳但贵
一句话：把「绝对打分」换成「组内相对名次」，省一个网络、天然稳，特别吃可验证奖励。
来源：GRPO / DeepSeekMath arXiv:2402.03300 · DeepSeek-R1 arXiv:2501.12948

26怎么选 + 三个最该拧的旋钮

一条决策线：冷启动 → 离线对齐 → 在线精修；难点永远在离参考多远、探索多少、学多快。

off-policy · 离线

  • 数据固定，不在线采样
  • 省、稳、易部署
  • 短板：OOD 弱、不探索
  • 用在：冷启动后快速对齐

on-policy · 在线

  • 自采样，RM 当奖励
  • 探索强、泛化强
  • 短板：贵、不稳、会 hack
  • 用在：要超越数据的精修
三个旋钮：① KL/β（离参考多远）· ② 探索量（组大小/采样数）· ③ 学习率——先动 KL，再动探索，最后调 LR。

27Agentic RL：把 RL 从「单轮回答」搬到「多步行动的轨迹」

单轮 RL 优化这次答得好不好；Agentic RL 优化在能动手的环境里，一整条「观察 → 思考 → 调工具 → 再决策」的轨迹能不能把事做成。

和单轮 RL 差在哪

  • 优化对象：单条 response → 多轮 episode
  • 奖励：即时偏好分 → 任务成败（稀疏、延迟）
  • 难点：答得好不好 → 长程 credit assignment（哪一步导致成/败）
  • 配套：cold start（先 SFT 学会工具格式）+ GRPO/PPO

奖励从哪来（两类）

  • 可验证奖励 RLVR：code/math/检索有客观对错，环境直接判
  • Rubric / PRM 过程分：医疗问诊等无唯一答案，给中间步打分
  • 即本文「打过程分」在 agent 场景的形态

RLHF 教模型怎么说；Agentic RL 教它在环境里一步步把事做成——search / coding / 医疗 agent 这条线要训的就是它。

28训练之外的轻杠杆（一）· PE 优化：不动权重，最快换行为

同一个模型，只改提示词就能换行为——零训练、分钟级上线、随时回滚。

1改什么

  • System prompt：把 Rubric 维度写成硬指令
  • Few-shot：喂 badcase 的正确做法
  • 输出格式：先结论后依据

2例子

  • 电商：先判软广，刷单评价不作推荐依据。
  • 医疗（Medprompt）：动态 few-shot + 自生成 CoT + 选项乱序集成，把通用模型在 MedQA 推过 90%。

3边界 / 坑

  • prompt 越长越贵、越不稳
  • 知识没沉进权重，模型没真变强
  • 易过拟合，要与 RM 对齐
PE 是最快验证"尺子怎么打"的手段：先用 prompt 试方向，再决定要不要训进模型。

29PE 激发出来的能力怎么收口：优先 RL/OPD，别默认 SFT

PE 能稳定激发 = 能力基模里本来就有（latent）。那是"让触发更可靠"的问题，不是"装能力"——用 SFT 烙印就容易训死。

1先问：要进权重吗

是流程/格式/工具 → 打成 Skill / 工具，别训。行为活在权重外，零训死、可热插拔。

2放大已有能力：激发类

  • RL（GRPO/PPO+rubric）：优化奖励不克隆 token，保泛化
  • OPD：学自己轨迹，不灌外来 token
  • 自蒸馏：留对的喂回，全程自分布

3真要 SFT：别训死

  • 只当 cold start，随后 RL/OPD 捞回泛化
  • LoRA：冻主干、不覆盖
  • 混 replay 数据、LR 低、轮数 1–2
收口：PE 试出来了 → 走激发（RL/OPD/小开关），不走烙印（SFT）；SFT 只留给基模确实不会、RL 又一直做不对的硬骨头。

30训练之外的轻杠杆（二）· Skill 优化：把专家流程打成能力包

比 PE 更进一步：把"某类任务该怎么一步步做"打包成可复用、可版本管理、按需加载的模块。

PE · 单条指令的临时改写

  • 改一句 prompt，解一类小问题
  • 难沉淀、难复用、难组合
  • 复杂多步流程一塞就乱

Skill · 结构化的能力单元

  • SOP + 工具 + 模板打成一包
  • 渐进式加载，用到才进上下文
  • 是 Agent 在循环里随手就调的能力
医疗（MedAgents / AMIE）：多专家会诊 / 多步问诊——分诊 → 鉴别诊断 → 管理推理；电商"导购选品 skill"：处境抽取 + 真实评价召回 + 适配打分 + 避坑清单。

31Harness 升级（一）· 从「写死的 workflow」到「模型自主调 skill」

Harness = 智能体的运行骨架。升级的本质：把流程控制权从开发者写死的编排交回给模型自己决策。

旧 · Workflow 编排

  • 把任务拆成固定有向图：A→B→C 写死
  • 每一步调哪个工具、什么顺序，代码里钉死
  • 模型只在格子里填空，不决定走向
  • 加一种情况 = 改图、加分支、回归测试

新 · Skill 化 Harness

  • 能力打成一个个 skill 包（SOP+工具+模板）
  • Harness 只给一圈 think–act–observe 循环
  • 模型自己判断该调哪个 skill、调几次、何时停
  • 加能力 = 丢一个新 skill 进去，不动主干
一句话：workflow 是「编导演」，skill 化 harness 是「给演员剧本库，让他自己演」。

32Harness 升级（二）· 优劣势权衡，与升级到底换来什么

不是 workflow 一无是处——它可控、可预测；但业务一复杂，写死的编排就脆、不泛化、组合爆炸。

Workflow · 优劣

  • + 路径确定、好调试、好兜底
  • + 弱基模也能跑，行为可审计
  • − 每个新场景都要重写一条线
  • − 分支越加越乱，改一处动全身
  • − 撞上没写到的情况直接抓瞎

Skill 化 Harness · 优劣

  • + 能力可组合，新场景靠已有 skill 拼
  • + 随基模变强自然涨，不重写主干
  • + 渐进加载，省上下文
  • − 吃基模能力，弱模型乱调、调飞
  • − 路径不定，调试 / 兜底更难
升级换来的：① 加能力 = 加 skill，边际成本趋零；② 主干稳定、能力解耦；③ 把决策权交给越来越强的模型，吃到基模红利。
落地节奏：高风险 / 强合规环节仍保留 workflow 护栏，低风险、需泛化的环节优先 skill 化——两者混用，不是非此即彼。

33训练之外的轻杠杆（三）· 知识层：给 Agent 一个「活的 wiki」

不训练、不改 prompt——把知识沉成 markdown 文件库，让 Agent 自己读、自己改、自己连。

RAG · 查询时现拼

  • 每次从原始文档切片召回
  • 反复读同样的事实
  • 无沉淀、易出错、费 token

LLM Wiki · 预先合成

  • 概念 = 1 个 md 文件，互相链接成图
  • Agent 读 / 改 / 连成「活的记忆」
  • 合成提前做，省 token、更准
Google 把它标准化为 OKF（Open Knowledge Format，2026-06）：md 目录 + YAML frontmatter（唯一必填 type，余 title/description/resource/tags/timestamp）+ index.md / log.md。
同源思路：Karpathy「LLM Wiki」gist · Obsidian vault · AGENTS.md / CLAUDE.md 约定文件。来源：Google Cloud, Introducing the Open Knowledge Format, 2026-06-13。

34训练之外的轻杠杆（四）· 记忆系统优化：让 Agent 自己沉淀最佳 Tool 与 Skill

35训练之外的轻杠杆（五）· 代码化：能写测试的，就别写进 prompt

PE / Skill / 知识层都在"不动权重"地塞行为——但塞进去的很多东西，其实是被错放成自然语言的代码。
prompt / skill 里的内容判据（怎么认）该沉到哪
流程 / 规则 / 阈值 / 格式 / 调度有明确答案、确定性、判对错不需语言代码 / 工具 / 校验器
领域知识 / 事实 / few-shot 示例是数据不是逻辑知识层（md / OKF）
意图理解 / 生成 / 语气 / 模糊判断 / 兜底判对错需要语言与语境留给模型——这是护城河
软措辞「建议 / 可考虑 / 个体化」像规则、实则模糊慎判：硬代码化会静默出错

推论：可代码化的越多 = 模型不可替代的越窄，但越关键；把算力烧在那条窄缝上，而不是背规则。

37Reward Hacking：模型会"骗尺子"，所以要对抗样本

长时间没法把 RM 调到理想态——它很容易被 hack。
Hacking 姿势表现
答案堆长用长度冒充丰富度
堆结构大量标题，信息密度其实很低
堆引用引用 / 截图不直接支撑结论（软广常用）
过度免责声明安全但难用
空泛正确的废话"因人而异、适合自己最重要"——正确但没用
解药：对抗样本 + 独立 GroundTruth + 线上 AB，三道关一起兜。

38后验校准：用户用脚投票，但同一个动作有两义

别只看后验：动机可能相反

  • 长停留：戳中认真读 / 太乱在反复翻找
  • 快划走：一眼解决 / 全是广告才划走
  • 追问：被启发 / 答非所问再问一遍
  • 点赞：真心认可 / 点赞抽奖

优势多目标：三者对齐

人标 / Rubric 定方向 → 线上行为验证 → 先验 (RM) 与后验 (行为) 同时改善。

危险信号：RM 离线一直涨、线上 AB 不涨——学到的是评测偏好，不是用户偏好。

线上 AB 必须拆到细分人群 / 品类才看得清，别掉进"全局指标"陷阱。

39GroundTruth 评测集：尺子到底对不对的最后兜底

它要包含什么

  • 用户真实场景 / 真实 query 分布
  • 高质量人工数据
  • 对抗样本（Code 样本）
  • 足够多样性（用户视角天然偏头部）

两个误区

  • 别拿评测集直接蒸馏 RM：会过拟合、对齐内部标准
  • 别只看全局指标
  • 正解：用评测集的度量方案去生成 RM，但不用评测集本身。
数据分工：机器合成训练 → 人工抽检调参 → 人工高质量评测 → Online AB 最终校准。

40刷榜：分数想刷多高都行——区别在「拟合榜单」还是「把榜当能力基建」

以现在的技术，单个榜理论上想刷多高都能刷到，各家只是相对克制；要同时刷很多榜，难度才陡增。

纯刷：直接拟合榜单

  • 拿榜直接拟合 / 套测试分布冲分
  • 不高级、不道德、用处也不大
  • 分数离开这个榜就不迁移

正解：把榜当能力基建

  • 把榜单背后的能力当成重要基础建设
  • 用真训练能力的方式做：SFT / RL（同前几节）
  • 分数提上去 = 内化成模型真本事
同一个分数，两种来历：拟合出来的一离开榜就废，训出来的走到哪都带得走。

41正向：从零定义一个场景的六步（以电商导购搜为例）

1 场景预期：抓处境标签（肤质 / 机型 / 预算 / 尺码 / 使用场景）

2 度量定义：复用通用维度 + 写场景 Rubric（刷单识别 / 适配档位）

3 评测集：贴合真实分布 + 高质量、能识破 reward hacking

4 训练过程：随机 session + badcase 专项 + 线上反馈，合成扩样本

5 训练 RM：pairwise（真实评价 vs 刷单软文）+ rubric 打分

6 在线校准：RM 预测对 AB 了吗？补 hard example、修 Rubric

换到医疗问诊搜：处境 = 年龄 / 基础病 / 用药史；Rubric = 红旗 / 剂量 / 就医提示；评测集要含高风险对抗样本；RM 偏好"该 abstain 就 abstain"。
核心：脑中先有"最优答案"，一步步推演——只做一步看一步，做不出超预期。

43总结：关键手段，与新人最容易踩的红线

四类关键优化手段

  • 维度评估标准准不准（尺子刻歪）
  • 维度在 RM 里的权重
  • Tool / 召回的原材料质量
  • 训练集——构造 hard example

新人六条红线

  • 先有度量定义，才有优化逻辑
  • 别用线性加权 merge
  • 别让合成数据自产自销
  • 别用评测集蒸馏 RM
  • 别只看全局指标
  • 警惕离线涨、线上不涨
